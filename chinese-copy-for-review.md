AEROS — 具身智能体的治理运行时

### Section 2: 过渡句
不止 Demo，要真正部署上去。

### Section 3: The Gap
- 标签: 行业缺口
- 标题: 模型会想了，机器人会动了，但智能体还是部署不出去。
- 描述: 模型越来越聪明，硬件越来越多，但真到了要上线的时候——谁来管 Agent 的生老病死？出了问题谁负责？升级能力会不会把线上搞崩？这些问题，现在没有人管。

**卡片 1:**
- 标题: 会规划 ≠ 能上线
- 正文: Agent 输出一个计划，不代表它能持续跑。状态丢了怎么办？任务卡住了怎么办？环境变了怎么恢复？这些靠模型解决不了。

**卡片 2:**
- 标题: 硬件越来越多，栈越建越重
- 正文: 人形、四足、AMR、协作臂——每种机器人都从零搭一套 Agent 软件？不现实。需要一个跑在硬件之上的通用运行时。

**卡片 3:**
- 标题: 要上线，就得管得住
- 正文: 仓库、医院、公共场所——Agent 的每个动作都得有人管。谁批准的？出了问题能回滚吗？全链路有记录吗？

### Section 4: How AEROS Works
- 标签: 运行机制
- 标题: AEROS 怎么工作
- 描述: 不再是跑一次就完事。AEROS 让每次执行都有准入、有监控、有兜底、有记录。
- 流程步骤:
  1. 用户目标
  2. Agent 规划
  3. (AEROS 接管)
  4. 能力准入
  5. 契约校验
  6. 策略检查
  7. 执行
  8. 监控巡查
  9. 恢复 / 回滚
  10. 审计日志
- 右侧解释标题: 每个动作都过一遍检查
- 右侧解释正文: Agent 想做什么都行，但 AEROS 会先问：这个能力允许跑吗？契约对得上吗？策略放行吗？谁在监控？出了问题怎么收场？
- 右侧解释正文 2: 前两步是 Agent 自己的事——随便哪个规划器都行。从第 3 步起，AEROS 说了算。
- 高亮句: 到不了硬件，除非先过运行时。

### Section 5: Three Core Layers
- 标签: 核心能力
- 标题: 三层架构，让 Agent 部署得出去
- 卡片 1: Runtime 核心 (Apache 2.0) — Agent 的执行底座。管生死、管调度、管状态、管执行。
- 卡片 2: ECM 能力包 (Apache 2.0) — 能力模块化打包，装上就能用，升级有契约，回滚有保障。
- 卡片 3: 治理层 (企业版) — 策略谁定的、动作谁批的、出了事谁负责——全链路管到底。
- 链接文字: 了解更多 →

### Section 6: Not Another Robot OS
- 标签: 定位
- 标题: 不是又一个机器人 OS
- 对比 1: ROS 2 解决**通信**问题 → AEROS 解决**治理运行时**问题
- 对比 2: 工作流工具 解决**编排**问题 → AEROS 解决**运行时约束下的持久执行**问题
- 对比 3: 模型栈 解决**推理**问题 → AEROS 解决**准入、监管、恢复和策略管控**问题
- 类比:
  - Linux 给机器一个系统层。
  - Kubernetes 给分布式应用一个控制层。
  - AEROS 给具身智能体一个治理运行时。

### Section 7: Use Case Preview
- 标签: 落地场景
- 标题: 为真实具身部署场景而生
- 卡片 1: 人形机器人 — 运行持久化具身 Agent，治理技能执行、生命周期管控、安全能力演进。

### Section 8: Five-Layer Architecture
- 标签: 架构
- 标题: 五层运行时架构
- 层级描述:
  - Layer 5: Agent — 应用逻辑 & 决策
  - Layer 4: Governance — 策略执行 & 审计
  - Layer 3: ECM Package — 能力模块 & 契约
  - Layer 2: Runtime — 生命周期 · 调度 · 执行
  - Layer 1: Embodiment — 硬件抽象 & 传感器
- 右侧标题: 治理，从第一行代码开始
- 右侧正文: 传统机器人中间件把治理当补丁——部署完了再加监控。AEROS 不一样。治理是运行时的一等原语，每次能力调用到达硬件前，都得先过治理引擎。
- 右侧正文 2: 结果：安全是架构保证的，不是靠祈祷。能力升级是受控的，不是听天由命。审计链路是完整的，不是抽样的。

### Section 2: Runtime Core (Agent 运行时核心)
- 徽章: Apache 2.0
- 标题: Agent 运行时核心
- 核心问题: 智能体在真实世界里怎么才能持续跑、稳定跑、挂了能恢复？
- 功能列表:
  - 生命周期管理 — 初始化、启动、暂停、恢复、停止、重启——完整的 Agent 状态机
  - 状态持久化 — 崩溃、重启、迁移都不丢状态
  - 任务调度 — 按优先级调度，支持抢占和 deadline 保证
  - 执行闭环 — 感知→决策→执行循环，带超时兜底
  - 故障恢复 — 四级递进恢复：重试 → 降级 → 回退 → 停机
  - 全链路追踪 — 原生 OpenTelemetry，每一步执行都可追溯

### Section 3: ECM Package System (ECM 包管理系统)
- 徽章: Apache 2.0
- 标题: ECM 包管理系统
- 核心问题: 智能体的能力怎么标准化、怎么版本管理、怎么安全升级？
- 功能列表:
  - ECM 标准格式 — 声明式能力模块，自带元数据、依赖关系和契约定义
  - 安装 · 升级 · 回滚 — 原子操作，出问题秒级回滚
  - 语义化版本 — 契约层面追踪哪些变更是破坏性的
  - 六维能力契约 — 感知、认知、行动、安全、资源、接口——六个维度全覆盖
  - 兼容性校验 — 装之前先检查：硬件够不够、运行时兼不兼容、策略允不允许
  - 影子模式 — 新能力先跑在旁边观察，不碰生产环境

## 技术架构页 (`/zh/technology`)

### 页面标题
技术架构 — AEROS

### Section 1: Header
- 标签: 技术架构
- 标题: 一台机器人，一个 Agent
- 描述: 能力靠 ECM 加载，行为靠治理约束。不搞多智能体集群，不搞模型直连硬件。一个 Agent 管到底。

### Section: Architecture Philosophy
**卡片 1:**
- 标题: 一机一 Agent
- 正文: 每台机器人对应一个持久化 Agent。身份、状态、学习经验，贯穿整个运行周期不丢失。

**卡片 2:**
- 标题: 能力可插拔
- 正文: Agent 本身不写死能力，而是按需加载。ECM 包独立安装、独立版本管理，随时升级和回滚。

**卡片 3:**
- 标题: 治理不是可选项
- 正文: 每次能力调用都得过治理引擎。安全靠架构保证，不靠理想主义。审计链路完整记录，不做抽样。

### Section: Five-Layer Architecture (Interactive)
- 标签: 架构
- 标题: 五层运行时架构
- 描述: 点击每一层，看看它负责什么。

**第 5 层：Agent**
- 描述: 最上层，跑决策逻辑。每个 Agent 都有自己的身份、记忆和目标，是个持久化的实体。
- 组件:
  - 决策引擎 — 对接大模型，负责规划和推理
  - 状态管理器 — Agent 状态跨会话、跨重启持久保存
  - 目标追踪器 — 拆任务、盯进度、必要时重新规划
  - 经验记忆 — 学到的行为、踩过的坑、适应性数据

**第 4 层：治理**
- 描述: 策略执行层。每次能力调用在真正执行前，都要过一遍当前生效的策略。
- 组件:
  - 策略引擎 — 声明式语言，写清运行边界
  - 监控器系统 — 实时盯住不变量，违规立刻触发
  - 恢复编排器 — 记录 → 告警 → 干预 → 停机，逐级响应
  - 审计记录器 — 每个决策和行动都有不可篡改的日志

**第 3 层：ECM 包**
- 描述: 能力层。Agent 的能力以模块形式加载，每个模块都有版本号和契约绑定，随时可装可卸可回滚。
- 组件:
  - 契约注册中心 — 六维类型化接口，连接 Agent 和硬件
  - 包管理器 — 装、升、回滚，全部原子操作
  - 兼容性校验器 — 装之前先看硬件和策略是否允许
  - 影子执行器 — 新能力先在旁边跑，不碰生产环境

**第 2 层：运行时**
- 描述: 执行层。管 Agent 的生老病死，调度任务，跑感知→决策→执行闭环，处理各种异常。
- 组件:
  - 生命周期管理器 — 完整状态机：初始化、启动、暂停、恢复、停止、重启
  - 任务调度器 — 按优先级调度，可抢占，保 deadline
  - 执行闭环 — 感知→决策→执行，带超时兜底
  - 故障处理器 — 重试 → 回退 → 降级 → 停机，分级应对

**第 1 层：具身层**
- 描述: 硬件抽象层。把五花八门的传感器、执行器和计算资源统一成标准接口。
- 组件:
  - 传感器抽象 — 相机、LiDAR、IMU、力/扭矩——统一数据格式
  - 执行器接口 — 臂、轮、爪、腿——统一控制 API
  - 资源监控器 — CPU、GPU、内存、电池——实时掌握资源状态
  - 硬件注册中心 — 自动发现设备，自动枚举能力

### Section: AEROS 与 ROS 2
- 标签: 定位
- 标题: AEROS 与 ROS 2
- 右侧标题: 不同层，不同问题
- 右侧正文 1: ROS 2 是通信中间件，管节点之间怎么传消息。AEROS 在 ROS 2 之上，管智能体怎么跑和怎么治理。
- 右侧正文 2: 打个比方：Linux 和 Kubernetes 的关系，就像 ROS 2 和 AEROS。Kubernetes 不是 Linux 的插件，而是在更高层解决不同的问题。AEROS 也一样。
- 对比:
  - ROS 2 管的 — 话题、服务、动作、节点、DDS 传输
  - AEROS 管的 — Agent 生命周期、ECM 包、治理策略、审计链路

- 副文: 不是又一个机器人 OS，也不是模型的包装层。我们要填的是"Demo 很酷"到"量产能用"之间那道鸿沟——让它变得可管理、可审计、真正安全。

### Section: Who We Are (我们是谁)
- 标签: 我们是谁
- 标题: 从研究到运行时
- 段落 1: AEROS 起步于一个系统研究计划——七篇论文，探索具身智能体真正可部署到底需要什么。不是"足够聪明"，不是"能动就行"，是真正可部署：持续运行、安全受控、有治理、在真实世界里。
- 段落 2: 我们发现，从"Demo 很酷"到"生产能用"之间的鸿沟，不是模型问题，也不是硬件问题。是运行时问题。所以我们自己造了一个。
- 段落 3: 今天，AEROS 是一个 open-core 运行时：Apache 2.0 核心开源，能力包管理系统持续完善，企业治理层逐步成型。我们在上海。
